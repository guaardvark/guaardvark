# Agent Proving Grounds — Live Test Findings (2026-05-14)

We just put the Guaardvark agent through `data/agent/files/agent_course.html`
(a 20-challenge HTML obstacle course for autonomous vision agents).
Here's what we learned. We're asking for suggestions on the open problems
at the bottom.

## Stack
- Local LLM: **Gemma4:e4b** (vision-capable, unified see+decide model).
- Coordinates: Gemma4's own `box_2d` output, normalized to 1000 (Google
  standard); display is 1000×1000 on :99.
- See-think-act loop in `backend/services/agent_control_service.py`,
  servo in `backend/services/servo_controller.py`.
- Memory: `AgentMemory` table + lesson distillation; unified recall
  through `memory_api._query_memories()` (just shipped).
- DOM-assisted clicking is **disabled** this session — the
  viewport→screen translation in `dom_metadata_extractor.py` was feeding
  Gemma4 bad coords. Vision-only path is the calibrated default.

## What works

1. **Lesson distillation end-to-end.** POST /api/lessons/start →
   3 positive pearls under the lesson_id → POST /api/lessons/end →
   Gemma4 distills the pearls in ~7s into structured JSON
   `{title, steps[3], parameters[1]}` and writes an `AgentMemory` row
   with `source=lesson_summary`, `importance=0.85`. Clean, fast, and
   the parameterization (`{contact_card_name}`) was inferred correctly.

2. **Belief tracker (Phase 4).** Fires on cross-domain pages
   when knowledge-file claims don't match the visible screen. On the
   proving grounds (which isn't Firefox/Reddit/YouTube), 14
   `belief_update` rows were written in 7 min for elements the compact
   knowledge expected.

3. **Stripping the knowledge files dropped belief noise to zero.**
   1847 → 210 lines across recipes.json + self_knowledge.md +
   self_knowledge_compact.md. Post-strip belief_update rate observed
   on the same proving-grounds session: 14 rows in 7 min → **0 rows
   in 1 min**.

4. **Click outcome semantics are split** (`target_found`,
   `click_issued`, `post_action_effect`, `reason` — all populated in
   `data/training/knowledge/servo_archive.jsonl`). Archive schema
   carries `raw_response`, `parse_path`, `detection_source`,
   `vision_config_source`, `vision_internal_width` per row.

5. **Stripped prompts gave noticeably better agent behavior.**
   Bloated knowledge: explicit verbose prompt required, agent tried
   spurious targets (`AGENT.PROVING.GROUNDS panel`,
   `FIRST CONTACT card`), 5 attempts to solve Ch1.
   Stripped knowledge: simple "Click the orange ENGAGE button" prompt
   worked, agent stayed on target, vision MIGRATED upward across
   iterations (y=438 → y=438 → y=398), Ch1 solved in 3 attempts.

## What's broken — open problems

### Problem 1 — Vision center-of-card pull / systematic y-bias

Gemma4 has a noticeable downward y-bias on small targets *inside*
visible challenge cards. Measured during Ch1 attempts:

| Target asked for          | Actual y | Gemma4 box_2d cy | Δ        |
|---------------------------|----------|------------------|----------|
| "orange ENGAGE button"    | ~372     | 419, 450, 428    | +47..+78 |
| "FIRST CONTACT header"    | ~240     | 383, 402, 405    | +143..+165|
| same after lean prompt    | ~372     | 438, 438, 398    | +66..+66, then HIT |

The model finds the right *card* but pulls the y-center toward the
middle of the visible card rather than the specific element. With the
stripped prompt it converged — first two clicks at y=438, third at
y=398 — suggesting some within-task self-correction is possible.

When the page is scrolled (Challenge 1 not at scroll=0), the bias is
much worse (~400 px off), as if the model is predicting against an
unscrolled layout.

### Problem 2 — Post-click verification disabled

`agent_control_service.py:778-787` has the comment
`# CLICK POST-VERIFY DISABLED 2026-05-12 — reverted to April 13
behavior. The 0.5s + 0.005 pixel-diff check added in commit 2cac26a
was producing false negatives on slow-launching apps (Firefox: 1-5s
to render)`.

Servo reports every issued click as `success=True`; the agent has no
mechanism to detect whether the world actually changed. On the
proving grounds, the page advanced after click 3, but the agent
didn't know, kept clicking, and the run was recorded as
`last_result.success=False, reason=timeout`.

### Problem 3 — Induction path gated by `last_result.success`

`backend/api/agent_control_api.py:_induce_candidate_recipe` requires
`last.success=True` to fire. Because Problem 2 keeps that flag False,
positive feedback after real successes never generates candidate
recipes. The strong-positive importance-boost path
(`_detect_strong_positive`) is wired correctly but downstream of the
same gate.

### Problem 4 — No `double_click` action type

`agent_control_service.py:741` routes only `click` and `right_click`.
Twin Tap and any other native-dblclick UI is unreachable; the JS
listener requires a true `dblclick` event within 400 ms, not two
sequential `click`s.

### Problem 5 — Knowledge files are domain-coupled

Before the strip, `self_knowledge_compact.md` declared Firefox/Reddit
elements (Firefox icon, Reply, Add a comment, Launch Anyway) as
present. The belief tracker contradicted every one of them on the
proving grounds. The stripped version drops the per-app inventory
and that fixes the immediate noise, but the underlying design
assumes one app context.

## Asks (we want your specific suggestions)

For each open problem below, please give concrete advice. Short, no
fluff — what would you actually change?

A. **Problem 1 — vision y-bias.** What would you try, given Gemma4
   has this center-of-card pull and a within-task convergence
   tendency? Multi-pass detection (zoom into the predicted region)?
   Prompt-level "report bounding box of the element, not the
   container" framing?

B. **Problem 2 — post-click verification.** What would you put back
   that wouldn't false-negative on slow renders? Polling-with-timeout
   for a `success_proof` description? Region-diff at the click site
   rather than full-screen pixel diff? Vision-asked verification
   ("did anything change in this area?")?

C. **Problem 3 — induction gate.** Decouple `_induce_candidate_recipe`
   from `last.success`? Add a separate "user marked succeeded" gate?
   Use the post-click verification result instead?

D. **Problem 4 — double_click.** Quick add. Where would you put it,
   and would you also add `triple_click`, `drag_to`, and `hover`
   while we're here?

E. **Problem 5 — domain-coupled knowledge.** Should knowledge files
   be scoped to URL patterns? Per-app overlays loaded by context?
   Or should we just keep one tactical core and add per-domain
   overlays via a separate AgentMemory `source`?

Anything else you'd flag that we didn't ask about — please surface it.
